# AGENTS.md

## 프로젝트 목표

- 주문/결제/재고 차감 흐름에서 발생하는 동시성 문제를 테스트로 재현한다.
- 비관적 락, 낙관적 락, Redis 분산락, 멱등키를 단계적으로 적용하고 비교한다.
- 단순 CRUD가 아니라 정합성 보장 흐름을 보여주는 실무형 백엔드 프로젝트로 만든다.

## 기술 선택 기준

- Java 21
- Spring Boot 3.x
- Gradle Groovy DSL
- PostgreSQL
- Spring Data JPA
- Flyway
- JUnit5
- Testcontainers
- Redis는 분산락 단계에서 추가한다.

## 패키지 구조 원칙

- 도메인 중심 패키지 구조를 사용한다.
- 각 도메인은 `domain`, `application`, `infrastructure`, `presentation` 하위 패키지를 가진다.
- `domain`에는 핵심 규칙과 상태 변경 로직을 둔다.
- `application`에는 유스케이스와 트랜잭션 경계를 둔다.
- `infrastructure`에는 JPA Repository, 외부 연동 구현체를 둔다.
- `presentation`에는 Controller, Request, Response를 둔다.

## DDD 적용 기준

- 과한 추상화는 하지 않는다.
- 처음부터 Hexagonal Architecture를 강하게 적용하지 않는다.
- Aggregate, Domain Event, Port/Adapter는 필요해지는 시점에만 도입한다.
- Entity에 모든 비즈니스 로직을 몰아넣지 않고, 도메인 규칙과 유스케이스 책임을 구분한다.

## TDD 원칙

- 도메인 규칙은 단위 테스트를 먼저 작성한다.
- 동시성, 트랜잭션, 락 전략은 통합 테스트로 검증한다.
- 실패하는 테스트를 먼저 만들고, 그 다음 구현한다.
- 동시성 테스트에는 `ExecutorService`, `CountDownLatch`를 사용한다.
- 테스트 이름은 어떤 상황에서 어떤 결과를 기대하는지 명확히 작성한다.

## 작업 방식

- 한 번에 큰 기능을 만들지 않는다.
- 1개 작업은 1개 목적만 가진다.
- 구현 전에 영향 범위를 먼저 설명한다.
- 작업 후 변경 파일 목록과 테스트 결과를 요약한다.
- 불필요한 포맷팅 변경을 하지 않는다.
- 새 파일은 필요한 경우에만 만든다.

## 주석 작성 규칙

### 클래스 주석

- 주요 도메인, 서비스, 유스케이스 클래스에는 클래스 역할을 설명하는 주석을 작성한다.
- 단순 DTO, Request, Response, Repository 인터페이스에는 불필요한 클래스 주석을 강제하지 않는다.

### 메서드 주석

- 비즈니스 의미가 있는 public 메서드에는 메서드 목적과 처리 흐름을 설명하는 주석을 작성한다.
- 단순 getter/setter, 단순 생성자, 단순 repository 메서드에는 주석을 강제하지 않는다.
- 테스트 메서드는 테스트 이름으로 의도가 충분히 드러나면 주석을 생략할 수 있다.

### 한 줄 주석

- 복잡한 조건문, 트랜잭션 경계, 락 처리, 멱등 처리, 재시도, 보상 처리에는 한 줄 주석을 작성한다.
- 코드만 봐도 명확한 단순 대입, 단순 if, 단순 return에는 주석을 달지 않는다.
- 주석은 "무엇을 하는지"보다 "왜 이렇게 처리하는지"를 우선 설명한다.

### 비즈니스 주석

- 주문 생성, 재고 차감, 결제 승인/실패, 재고 복구, 멱등키 검증, 동시성 제어 같은 핵심 비즈니스 흐름에는 단계별 주석을 작성한다.
- 예외가 발생하는 조건에는 비즈니스 관점의 이유를 주석으로 남긴다.
- 동시성 테스트에서는 어떤 경합 상황을 재현하는지 주석으로 설명한다.

### 주석 스타일

- 주석은 한글로 작성한다.
- 과하게 장황하게 쓰지 않고, 실무자가 읽었을 때 흐름을 빠르게 이해할 수 있게 작성한다.
- 코드와 맞지 않는 오래된 주석이 남지 않도록, 로직 수정 시 주석도 함께 수정한다.
- TODO 주석은 반드시 이유와 후속 작업 방향을 함께 적는다.
- 좋은 주석 예: 재고 차감은 주문 확정 전에 수행해 overselling을 먼저 차단한다.
- 나쁜 주석 예: quantity를 감소시킨다.

## PR 작성 규칙

- PR 제목과 본문은 기본적으로 한글로 작성한다.
- PR 제목은 변경 목적이 드러나게 짧고 명확하게 작성한다.
  - 예: 프로젝트 작업 규칙 문서 추가
  - 예: Spring Boot 기본 골격 구성
  - 예: 재고 차감 동시성 테스트 추가
- PR 본문에는 아래 항목을 포함한다.
  - 변경 내용
  - 검증 결과
  - 참고 사항 또는 남은 작업
- 테스트를 실행하지 못한 경우에는 실행하지 못한 이유를 명확히 작성한다.
- 구현되지 않은 내용을 PR 본문에 과장해서 쓰지 않는다.

## 금지 사항

- 처음부터 Kafka, RabbitMQ, MSA를 도입하지 않는다.
- 처음부터 Redis 분산락을 적용하지 않는다.
- H2만으로 동시성 검증을 끝내지 않는다.
- 구현되지 않은 내용을 README에 과장해서 쓰지 않는다.
- 기능과 무관한 리팩토링을 하지 않는다.
